软件测试与质量初始

软件测试与质量初始

二月 23, 2020

本章内容：

一、概述

  1. 软件测试的意义

    1. 为什么要软件测试
      1. 软件测试职位本来带来的价值：确保产品质量
      2. 软件测试人员比其他人做软件测试测试工作，效率更高，效果更好
      3. 软件测试是软件产品质量保证的重要措施之一
    2. 哪些人应了解软件测试
      1. 用户
      2. 项目经理
      3. 程序员
      4. 测试员

  2. 软件测试能力成熟度模型

    1. 初始阶段（软件测试）

      目标 ：表明软件是奏效的

    2. 定义阶段（软件缺陷）

      目标：表明软件符合其技术规范

    3. 集成阶段

      考虑客户和用户需求来建立测试目标，成立专门的测试组织

    4. 管理与测量

      引入审评机制，管理测试用例和测试过程

    5. 最佳化

      建立缺陷思想，开展与测试有关的度量活动

  3. 2018年测试工具小结：

    1. 功能测试: Selenium / Appium
    2. 性能测试: Jmeter
    3. 测试管理: 禅道, Jira
    4. 单元测试: JUnit , PyTest
    5. 行为驱动开发: Cucumber

二、软件测试的定义

  1. 软件测试

    • 软件测试定义：软件测试是人工地，或通过使用工具来自动地运行被测软件系统，或静态检查被测系统的过程。
    • 目的：在于校验被测系统是否满足需求， 或要弄清楚实际的系统输出与预期系统输出之 间的差异。
    • 定义需求（委托方）–分析需求（双方）–实现需求（开发人员）–校验需求（测试人员）

  2. 软件测试的基本手段：

    • 设计测试用例

  3. 软件测试的两种测试策略

    1. 动态测试
      1. 手工动态测试（文件：测试用例，驱动/桩模块，缺陷报告）
        • 提供被测对象
        • 准备用户需求
        • 搭建测试环境
        • 设计测试用例
        • 运行测试用例
        • 检查测试结果
        • 记录测试过程
        • 报告发现的缺陷
        • 执行回归测试
      2. 自动化动态测试（文件：测试用例，驱动/桩模块，测试脚本，缺陷报告）
        • 提供被测对象
        • 准备用户需求
        • 搭建测试环境
        • 设计测试用例
        • 编写测试脚本
        • 运行测试用例
        • 检查测试结果
        • 记录测试过程
        • 报告发现的缺陷
        • 执行回归测试
    2. 静态测试（测试用例，缺陷报告）
      • 提供被测对象
      • 准备用户需求
      • 阅读代码
      • 阅读文档
      • 报告发现的缺陷
      • 执行回归测试

  4. 软件测试的两种执行方式

    1. 人工
    2. 自动化

  5. 软件测试的一般流程

    1. 设计

      1. 输入

        • 需求规格说明
        • 项目计划

      2. 输出

        • 测试计划

          谁，何时，使用什么方法，用到什么资源，遵循什么标准，对什么展开测试，可能有哪些风险？

    2. 计划

      1. 输入
        • 需求
        • 设计文档
        • 测试计划
      2. 输出
        • 测试用例
        • 测试过程

    3. 实施

      1. 输入
        • 测试用例
        • 测试过程
        • 需求
      2. 输出
        • 测试驱动模块
        • 测试桩模块
        • 测试脚本（如果是自动化测试应有此）

    4. 评估

      1. 输入
        • 测试用例
        • 缺陷报告
        • 测试标准
      2. 输出
        • 测试评估报告

三、软件缺陷

  1. 站在用户角度看软件

    1. 1945.09.09 Grace Hopper 计算机之母 “First actual case of bug being found”

    2. 用户角度定义缺陷：

      软件测试员认为软件难以理解、不易使用、运行速度缓慢、或者最终用户认为不好，则是缺陷；

    3. 为减少此类缺陷：

      • 对需求进行书面约定：需求规格说明
      • 必须确保需求的稳定

  2. 站在开发方角度看软件

    1. 开发方角度定义缺陷：

      1. 软件未达到需求规格说明书中指明的功能，则是缺陷；

        为减少此类缺陷：

        • 检查正常功能、正常流程

        • 检查性能

          1. 软件出现了需求规格说明书中指明不会出现的错误，则是缺陷；

        为减少此类缺陷：

        • 检查异常情况

        • 检查无效用户输入的识别能力

        • 检查无效用户输入的处理能力

          另外：

        • 测试效果严重依赖于测试人员的经验

          1. 软件功能超出需求规格说明书中指明的范围，则是缺陷；

        造成缺陷的原因：

        • 无意加入，过错缺陷

        • 人为加入，需求缺陷

        • 人为加入，过错缺陷

        • 人为加入，病毒

          1. 软件未达到需求规格说明书中虽未指出但应达到的目标，则是缺陷；

        造成缺陷的原因：

        • 隐含特性，需求缺陷

四、测试用例的定义

  1. 测试用例

    1. 测试用例的定义：
      • 能代表需求的小的测试单元
      • 描述用户预期输出
      • 反映系统实际执行结果
    2. 设计原则：
      • 基于风险最低、效率最高、分而治之的测试设计原则；
    3. 测试用例的组成：
      • 输入
        • 测试数据
        • 操作步骤
      • 输出
        • 系统预期执行结果
      • 测试环境（系统环境设置）
        • 工作平台
        • 前提条件
    4. 需要关注的问题：
      • 哪里是系统输入输出中最可能潜伏缺陷的地方
      • 以怎样的流程操控系统更有利于发现缺陷
      • 如何保证测试用例对系统的全覆盖
    5. 测试用例的基本属性：
      • 典型性
      • 可测试性
      • 可重视性
      • 独立性
    6. 测试用例的设计：
      1. 输入数据
        • 正常数据
        • 错误数据
          • 满足数据类型，不在有效取值范围内
          • 不完全满足数据类型
          • 输入条件缺失
        • 边界数据
      2. 操作步骤

五、软件测试分类

  1. 软件测试的不同分类方式

    1. 测试阶段的角度（对象角度）
      1. 单元测试：查找单元模块错误
      2. 集成测试：查找接口错误
      3. 系统测试：系统行为是否符合需求，系统与硬件设备、运行环境、应用程序之间的接口
      4. 验收测试：由系统提供者和客户共同完成
    2. 测试技术的角度
      1. 黑盒测试：关注与产品外部行为相关的缺陷
      2. 白盒测试：关注与代码内部结构相关的缺陷
      3. 灰盒测试：综合黑盒和白盒
    3. 测试目标的角度
      1. 功能测试：针对软件功能，检查是否符合预期
      2. 性能测试：是否满足性能需求
      3. Alpha测试(α测试)：由开发机构人员来模拟潜在用户开展测试
      4. Beta测试(β测试)：把早期版本发布给具有代表性的用户群来测试
      5. 回归测试：软件版本修改后的重新测试
      6. 压力测试：最大负载或超过最大负载运行软件，来确定系统运行的负载界限
      7. 负载测试：在资源超负荷的情况下发现设计上的错误或验证系统的负载能力
      8. 安全性测试：用于测试系统在遭遇未授权访问、计算机犯罪和破坏是能否保护自己
      9. 配置测试：对硬件配置和软件配置进行测试，以检测系统行为是否符合规格要求
      10. 安装测试：在目标环境中进行软件的安装
      11. 可用性测试：用于评估系统使用的简易程度，交互是否 具有人机工程学设计以及用户文档使用的有效性
      12. 可恢复性测试：：用于检验系统在灾难或意外宕机后的重 启能力
      13. …
    4. 测试执行方式的角度
      1. 手动测试：人工执行测试
      2. 自动化测试：节省人力，实践或硬件资源，并提高测试效率
      3. 半自动化测试

六、解读软件测试与软件质量

  1. 软件质量的定义

    1. ISO8402关于质量的定义：

      反映实体满足明确或隐含需要能力的特性总和；

      • 在合同环境中，需要是规定的，而在其他环境中，隐含需要 应加以识别和确定；
      • 在许多情况下，需要会随着时间而改变，要求定期修改规范；

    2. 质量的要素：

      • 实体：
        • 实体是质量的主体，即测量的主体
        • 一般指产品，也可以是活动、过程、组织、人，或者它们 的组合
      • 特性集合：
        • 表示度量
        • 特性必须可以测量，定性或定量方式得到指导性结论
      • 需求：

    3. 软件质量的定义：

      • 反映软件满足明确或隐含需要能力的特性总和
        • 客观而言，软件质量是软件具有某种能力的属性， 这是前提条件
        • 主观而言，软件具有的能力对应不同层次的用户 需求
      • 狭义的软件质量：软件的内部质量，即软件无“故障”
      • 广义的软件质量：产品质量、过程质量和客户满意度

    4. 不同层次的用户需求：

      • 显式需求
        • 需求规格说明书描述的内容
        • 是软件内部质量
      • 隐式需求
        • 未在需求规格说明书中明确描述
        • 用户明确说明的目标
        • 反映验收质量
      • 实际需求
        • 软件的使用质量
        • 用户在实际使用过程中对产品的质量评价

  2. 软件测试与软件质量的关系

    1. 软件测试可以验证软件质量
    2. 软件测试不能提高软件的质量
    3. 软件测试是软件质量保证的关键步骤
    4. 提高软件质量依赖于改进软件开发过程质量
      1. 曲棍球式管理模式：定期检查，发现问题再解决问题，不能杜绝问题
      2. 芭蕾舞式管理模式：改进开发过程，避免缺陷植入
